The Offramp API converts USDT to LKR and pays it into a Sri Lankan bank account. It is built for aggregators and exchanges that need to disburse funds to their own end-users programmatically.

For the endpoint reference and a live playground, see the **Offramp** endpoints in the sidebar.

<Warning>
  Offramp is available only to merchants with the `AGGREGATOR` role. Contact [support@ceypay.io](mailto:support@ceypay.io) to enable it for your account.
</Warning>

---

## Transfer Flow

<Steps>
  <Step title="Look up bank codes">
    Call `GET /v1/bank/list` for CEFTS bank codes. This is static reference data — fetch it once and cache it.
  </Step>
  <Step title="Register the end-user">
    Call `POST /v1/aggregator/user` with your own user ID. The call is an upsert, so it is safe to run before every transfer.
  </Step>
  <Step title="Register their bank account">
    Call `POST /v1/aggregator/user/{id}/bank-account` and store the returned `userBankId`.
  </Step>
  <Step title="Lock an FX rate">
    Call `GET /v1/aggregator/quote` with the USDT amount. The returned `fxLockId` is valid for 60 seconds.
  </Step>
  <Step title="Initiate the transfer">
    Call `POST /v1/aggregator/offramp` with the `fxLockId`, `userId`, `userBankId`, and your `externalRef`.
  </Step>
  <Step title="Track completion">
    Poll `GET /v1/aggregator/offramp/{id}`, or supply a `webhookUrl` to be notified instead.
  </Step>
</Steps>

Steps 2 and 3 run once per end-user and bank account. Both IDs are reusable across every subsequent transfer.

---

## Privacy

CeyPay stores only the `externalUserId` you supply, plus the bank account details needed to make the transfer. No other personal or identifying information about your end-users is collected or retained.

---

## FX Rate Locks

A quote locks the USDT/LKR rate for **60 seconds**. Submit the offramp before `expiresAt` — expired locks and locks already spent on another transfer are both rejected with a `400`.

Fetch the quote immediately before you create the offramp, not at the start of a longer user flow. If a lock expires, request a fresh quote rather than retrying with the old ID.

---

## Idempotency

`POST /v1/aggregator/offramp` is idempotent on `externalRef`. Resubmitting the same key — after a network timeout, for instance — returns the original offramp unchanged instead of creating a second transfer.

Use a stable key drawn from your own system:

```json
{ "externalRef": "withdrawal-9876543" }
```

Never generate the key randomly per attempt, and never reuse a key across genuinely different transfers.

## Offramp Statuses

| Status       | Meaning                                                       |
| ------------ | ------------------------------------------------------------- |
| `PENDING`    | Created and queued for processing                             |
| `PROCESSING` | Bank transfer in flight                                       |
| `COMPLETED`  | Settled — `bankRef` and `completedAt` are set                  |
| `FAILED`     | Could not be settled — the LKR is refunded to your float       |

A failed offramp posts a `REFUND` entry back to your float ledger, so your balance reconciles without manual intervention.

## Webhooks

Set `webhookUrl` on the offramp request to be notified when the transfer settles or fails. Offramp webhooks use the same ED25519 signature scheme as every other CeyPay webhook — see the [Webhooks Guide](/api/v1/webhooks) for verification details.

### payment.completed

```json
{
  "event": "payment.completed",
  "paymentId": "c1d2e3f4-a5b6-4c7d-8e9f-001122334455",
  "externalRef": "withdrawal-9876543",
  "amountLkr": 295500.00,
  "bankRef": "BOC-TX-123456",
  "failureReason": null,
  "completedAt": "2026-05-28T10:05:00.000Z",
  "failedAt": null
}
```

### payment.failed

```json
{
  "event": "payment.failed",
  "paymentId": "c1d2e3f4-a5b6-4c7d-8e9f-001122334455",
  "externalRef": "withdrawal-9876543",
  "amountLkr": 295500.00,
  "bankRef": null,
  "failureReason": "Invalid account number",
  "completedAt": null,
  "failedAt": "2026-05-28T10:05:00.000Z"
}
```

### Headers

```
X-Webhook-Signature: <base64-encoded ED25519 signature>
X-Webhook-Timestamp: 1748430300000
X-Webhook-Attempt: 1
Content-Type: application/json
User-Agent: CeyPay-Webhook/2.0
```

### Retry schedule

| Attempt | Delay after previous |
| ------- | -------------------- |
| 1       | Immediate            |
| 2       | 1 minute             |
| 3       | 2 minutes            |
| 4       | 4 minutes            |
| 5       | 8 minutes            |

CeyPay retries on connection timeouts, non-2xx responses, and network errors. Return `200` immediately and process the event asynchronously.

## Reconciliation

Two read-only reports let you reconcile against your own ledger. Both are paginated and support date-range filtering.

**Float balance ledger** (`GET /v1/aggregator/report/ledger`) — every LKR movement on your float: admin top-ups (`CREDIT`), offramp charges (`DEBIT`), and failed-transfer reversals (`REFUND`). The summary block carries opening and closing balances for the window, so a period reconciles end to end.

**Offramp transactions** (`GET /v1/aggregator/report/offramp`) — one row per offramp with the end-user ID and destination bank details included, so the export stands alone. The summary always covers the full date window even when you filter the rows to a single status, which keeps per-status breakdowns visible.
